# LessAI 分块测试（Markdown）

> 用途：尽量覆盖常见 Markdown 结构，并观察：
> 1) chunk 边界是否稳定；
> 2) 标题是否按设置受保护；
> 3) 行内保护区（如 `$...$`、`` `code` ``、链接）是否在 chunk 内高亮而不切断 chunk。

## 0. 阅读说明

- 建议先把粒度切到“空白分段”，看段落 chunk 是否符合直觉。
- 再切到“整句/小句”，看边界是否只在段落内部变化。
- 最后跑一次改写，观察：列表骨架（`- `、`1. `、`> `）是否被保持；保护区是否被锁定。

## 1. 段落与整句

第一段：这是一段包含多个整句的正文，用来检查整句切分是否稳定。数值 3.14 不应被当成句号边界，因此“3.14”应留在同一整句里。最后一句带问号，用来测试：整句切分是否在“？”处生效？

第二段：这里包含中英混排（e.g. / i.e. / U.S.A.），以及中文引号“引用内容”。如果你发现整句模式把 e.g. 切碎，这是边缘情况，可以记录下来。

第三段：这里插入一个明显的行内公式 $E=mc^2$，以及一个带反斜杠的表达 $\alpha+\beta\to\gamma$。
同时还有行内代码 `cargo fmt --check` 和 `E:\\Code\\LessAI\\testdoc`。
这些片段应该被标成保护区（不可改写），但不应切断 chunk 的淡紫色高亮。

## 4. 表格（常见易碎结构）

| 场景 | 你希望看到的效果 | 备注 |
| --- | --- | --- |
| 空白分段 | 只在空白行处分段 | 段落粒度最自然 |
| 整句 | 句号/问号等切分 | 适合长段落 |
| 小句 | 逗号/分号等切分 | 审阅更细但更碎 |

表格后面一句话：如果表格被当作“保护区”，chunk 仍应保持连续可读。

## 5. 代码块（多行保护区）

```bash
# 也包含看起来像 URL 的内容（应原样）

```

```json

  "purpose": "chunk and diff review",
  "numbers": [3.14, 2718.28]
}
```

结尾一句：代码块之后的正文应继续正常分块。

## 7. 数学与 `$` 歧义（优先保护但不要误伤）

这一段专门用来验证 `$...$` 的保护规则是否足够稳健：

- 行内数学：$E=mc^2$、$\alpha+\beta\to\gamma$、$\sum_{i=1}^{n} x_i$。
- 显示数学（`$$...$$`）：通常应当被整体保护。
- 美元符号（不是数学）：价格是 $5.00 或 $12.34；这里的 `$` 并不一定代表公式。
- 转义美元符号：写成 `\\$`（在 Markdown 中表示字面量 `$`），不应误触发数学保护。
- 缺失闭合的 `$`：例如这句有一个 `$` 但没有结尾美元符号（用于观察容错，不要求必须识别为数学）。

下面是一个典型的显示公式块：

$$
\mathrm{KL}(p\|q) = \sum_{i} p_i \log \frac{p_i}{q_i}
$$

公式前后各有一段自然语言：理想情况下，它们仍属于同一个 chunk（至少在“空白分段”下应连续可读），只是公式片段在 chunk 内高亮为保护区。

## 8. 内联代码与反引号（不要把代码当数学）

行内代码应整体保护：`cargo fmt --check`、`$HOME`、`E:\\Code\\LessAI\\testdoc`。

这里有一个“包含反引号”的代码展示：`` `literal backtick` ``（注意：这段使用双反引号包裹）。

还有一个故意混合的例子：在一句话里同时出现 `$E=mc^2$` 与 `` `$HOME` ``，
它们都应该高亮为保护区，但类型不同（数学 vs 代码），并且不应切断 chunk 高亮。
