LessAI 综合测试（纯文本 .txt）
生成日期：2026-04-09

目标：
- 用一个纯文本样本覆盖标题、正文、列表、链接、代码样排版、公式样排版、长段落和写回敏感区。
- 验证同一份正文在“段落 / 整句 / 小句”三种粒度下的切块是否稳定。
- 验证只改一个字、一个标点或一处空行时，diff 是否只影响局部。

建议观察顺序：
1. 导入文件后分别切到“段落 / 整句 / 小句”。
2. 每次切换后执行一次“重置记录”。
3. 开启高亮，确认纯文本没有额外锁定区，所有可读正文都能参与处理。

────────────────────────────────────────────────────────
1. 文档说明与测试目标
────────────────────────────────────────────────────────
这是一份只包含纯文本的综合样本，不混入 Markdown、TeX 或 Word 语法。它的用途不是展示格式，而是验证纯文本导入后，正文是否可以完整阅读、稳定分块、局部修改、正确写回。

建议先看每个章节标题是否和正文分成合理的块，再随机选一句话做微调，例如把“稳定”改成“更稳定”，确认变化不会扩散到整章。

────────────────────────────────────────────────────────

────────────────────────────────────────────────────────

下一段继续模拟报告正文。这里放入一个常见写法：先描述背景，再给出判断，最后补一句限制条件。小句模式下，这种长句通常会被切得更细。

3. 标点边界与数字边界
────────────────────────────────────────────────────────
这一段包含小数 3.14、带千分位的 2,718.28、日期 2026-04-09、版本号 v2.3.7，以及编号 No. 5。它们都不应因为句点、逗号或短横线而被误切成碎片。

这里再补一个中英混排场景：Dr. Wang 在 U.S.A. 完成实验后，于 2025-12-31 提交了 draft v1.0.2。若整句模式把缩写切得过碎，可以记录问题，但段落级阅读不应被破坏。

────────────────────────────────────────────────────────
5. 链接与可读正文
────────────────────────────────────────────────────────
正文里出现完整 URL 是很常见的，例如 https://example.com/docs/lessai?tab=writeback#docx。理想情况是阅读时不要把链接切碎，改写时也不要误删参数。

这一段还包含邮箱 foo.bar+tag@example.com 和备用网址 www.example.org/help。它们都应作为普通正文的一部分参与显示，但局部修改时不应导致整段 diff 扩散。

────────────────────────────────────────────────────────
6. 强调表达与常见样式的纯文本近似
────────────────────────────────────────────────────────
纯文本没有真正的加粗、斜体或下划线，所以这里只用词语顺序和重复表达来模拟强调：重点、非常重点、尤其需要重点关注。

还可以观察引号和括号的连续性：他说“保持稳定”，我补充“保持可读”，然后在同一句里插入（局部说明）继续向下读。小句模式下，逗号和分号会更积极地制造边界。

────────────────────────────────────────────────────────
7. 代码、命令与原样内容
────────────────────────────────────────────────────────
下面是纯文本形式的命令和伪代码，它们仍然是普通正文，但应尽量保持原样：

pnpm install
pnpm run tauri:dev
cargo test

if score >= 90:
  level = A
else:
  level = B

这里只测纯文本对齐，不要求把它识别成真正代码块；但局部改写时，不应把命令里的短横线、空格或冒号改坏。

────────────────────────────────────────────────────────
8. 数学与公式相关内容
────────────────────────────────────────────────────────
这里放一些正文中的公式样表达：E = mc^2，x_i -> x_bar，a <= b，n -> ∞，P(A|B) = 0.42。它们都应被当作普通文字处理，而不是额外保护区。

再放一组 Unicode 数学字符：α、β、γ、Σ、≤、≈、∞。导入后应能正常显示，改写后不应莫名变成问号或乱码。

────────────────────────────────────────────────────────
9. 表格样排版与非正文骨架
────────────────────────────────────────────────────────
下面是纯文本对齐的“表格样排版”，主要用于观察等宽内容是否会在编辑后错位：

项目        期望行为          备注
标题        可读可分块        不存在真正样式
链接        作为正文显示      不要切碎参数
长句        可在整句内切分    段落级仍要连续

这一类结构在纯文本里本质上仍是正文，所以不应被单独锁定；但如果你只改动一格里的一个词，diff 也不应波及整块。

────────────────────────────────────────────────────────
10. 混合段落与结构穿插
────────────────────────────────────────────────────────
这一段把多种元素放在一起：先写一个链接 https://example.com/api，再写一个命令 git status，然后写一句公式样表达 y = ax + b，最后补一段引号“这只是正文，不存在额外保护区”。

如果你在这段里只把“正文”改成“普通正文”，理想效果是只有附近一小块变化，而不是从链接一路扩散到整段末尾。

────────────────────────────────────────────────────────
11. 写回敏感区
────────────────────────────────────────────────────────
请只做下面三种最小改动之一：
1. 把这一句里的一个逗号，改成分号。
2. 在下一句前面插入一个空白行。
3. 把“最后一句？”改成“最后一句！”。

这是下一句，用来测试空行边界。
这是最后一句？

如果写回后重新导入，内容顺序、空行位置和相邻段落边界都应保持可预期。

────────────────────────────────────────────────────────
12. 长段落与压力观察区
────────────────────────────────────────────────────────
这一整段故意写得比较长，用来观察长正文在三种粒度下的切块稳定性：当一段文字同时包含中文叙述、English phrases、年份 2024-2026、缩写 e.g.、i.e.、小数 0.618、邮箱 qa-team@example.com、链接 https://example.com/report/final、命令 pnpm run build、以及括号（例如这里的补充说明）时，段落模式应优先保证整体可读，整句模式应主要在句号、问号、感叹号附近分开，小句模式则可以在逗号、分号、冒号附近进一步细分，但任何一种模式都不应把一个短链接、一个数字或一个缩写错误拆成大量无意义的小块。
