# LessAI 测试指南

## 1. 目标

这份文档用于覆盖当前仓库的主要功能：导入、切块、审阅、批处理、多选处理、手动编辑、选区改写、导出、写回，以及不同文档格式的支持边界。

建议按“自动化检查 -> 手工回归 -> 格式专项”三层执行。测试素材统一放在 [testdoc](../testdoc/)。

## 2. 准备

- 环境：Node 20+、pnpm 10+、Rust stable、Tauri 运行依赖。
- AI 相关功能需要真实可用的 OpenAI 兼容接口：`Base URL`、`API Key`、`Model`。
- 启动应用：

```bash
pnpm install
pnpm run tauri:dev
```

可直接使用的样例文件：

- `testdoc/chunk-test.txt`
- `testdoc/chunk-test.md`
- `testdoc/chunk-test.tex`
- `testdoc/chunk-test.docx`
- `testdoc/04-3 作品报告（大数据应用赛，2025版）模板.docx`
- `testdoc/chunk-test.pdf`

## 2.1 综合样本章节地图

`chunk-test.txt`、`chunk-test.md`、`chunk-test.tex`、`chunk-test.docx` 现在统一采用 12 节骨架，便于逐节人工核验：

1. 文档说明与测试目标
2. 标题层级与普通段落
3. 标点边界与数字边界
4. 列表与段内换行
5. 链接与可读正文
6. 行内强调与常见样式
7. 代码/命令/原样内容
8. 数学与公式相关内容
9. 表格或非正文结构
10. 混合段落与锁定区穿插
11. 写回敏感区
12. 长段落与压力观察区

每打开一个 `chunk-test.*`，都建议按这四个问题逐节核对：

- 这一节该不该分块。
- 这一节该不该锁定。
- 这一节是否允许编辑。
- 改后能否安全写回。

## 3. 自动化检查

每次发版前至少执行一次：

```bash
pnpm run typecheck
node scripts/ui-regression.test.mjs
cd src-tauri && cargo test
```

如果本轮改动涉及 docx 或“处理所选”，再补这两个定向检查：

```bash
cd src-tauri && cargo test docx -- --nocapture
cd src-tauri && cargo test rewrite_targets -- --nocapture
```

通过标准：

- `typecheck` 无报错。
- `ui-regression` 输出 `[ui-regression] OK`。
- `cargo test` 无失败。
- `docx` 定向测试应覆盖列表、表格占位、公式锁定、分页符、超链接、常见行内样式、嵌入对象拒绝。
- `rewrite_targets` 应覆盖“处理所选”只处理选中块、忽略不可改写块、索引越界报错。

## 4. 手工主流程回归

### 4.1 基础导入与切块

1. 打开 `testdoc/chunk-test.txt`。
2. 在设置里切换 `小句 / 整句 / 段落`。
3. 每次切换后点击“重置记录”。

期望结果：

- 文档能正常打开。
- 切段策略切换后，chunk 数量明显变化。
- 一旦存在修改对或生成进度，切段策略会被锁定；重置后才可重新生效。

### 4.2 人工把关模式

1. 模式设为“人工把关”。
2. 点击“开始优化”。
3. 在右侧对生成结果执行“应用 / 忽略 / 删除 / 重试”。

期望结果：

- 每次只生成下一段。
- 应用后全文视图能看到终稿变化。
- 删除后该段回到待生成状态。
- 重试只影响当前段，不影响其他段。

### 4.3 自动批处理模式

1. 模式设为“自动批处理”。
2. 点击“开始批处理”。
3. 依次测试“暂停 / 继续 / 取消”。

期望结果：

- 自动连续处理多个 chunk。
- 暂停后不再继续生成。
- 继续后恢复。
- 取消后保留当前已有结果，不清空进度。

### 4.4 多选批处理

1. 在文档区按住 `Ctrl / Cmd` 点击多个可改写块。
2. 确认内容区右上角出现“已选 N 段”。
3. 确认主按钮文案变为“处理所选”。
4. 分别在人工把关和自动批处理模式下执行一次。

期望结果：

- 只处理选中的 chunk。
- 不选中时，按钮文案恢复为“开始优化”或“开始批处理”。
- “已选 N 段”显示在内容区右上角，不挤压其他工具栏组件。

### 4.5 导出、写回、恢复

1. 在 `.txt` 上应用几条修改后点击“导出”。
2. 再点击“覆盖原文件并清理记录”。
3. 重新打开同一个文件。

期望结果：

- 导出生成新的 `.txt` 文件。
- 写回只写入已应用的修改，成功后清空该文档历史记录并重新打开。
- 同一文件再次打开时，若未 finalize/reset，应自动恢复会话。

## 5. 编辑模式与选区改写

使用 `testdoc/chunk-test.txt` 或简单 `docx`：

1. 在没有修改对、没有运行任务的前提下进入“编辑终稿”。
2. 手动改几处文本，测试“保存”“保存并返回”“放弃修改”“直接返回”。
3. 在编辑器内选中一段正文，执行选区处理。

期望结果：

- 有未保存修改时，不能直接离开编辑页、打开其他文件或直接开始 AI 优化。
- “保存”写回原文件。
- “保存并返回”回到工作台且保留最新正文。
- 选区处理只替换选中的文本，不影响其余内容。
- 选区过长时会弹风险提示，不会自动偷偷拆分。

## 6. 格式专项检查

### 6.1 Markdown

打开 `testdoc/chunk-test.md`，开启高亮/标记。

重点检查：

- 标题是否按“导入时标记标题为不可改写”设置生效。
- Front Matter、表格、HTML 注释、代码块是否整体锁定。
- 行内公式、代码、链接、图片语法、脚注引用、HTML 标签仍可见，但应作为保护区展示。
- 混合段落中只改正文时，不应破坏 Markdown 语法骨架。

### 6.2 TeX

打开 `testdoc/chunk-test.tex`。

重点检查：

- `\section`、`\subsection`、`\par`、列表环境、`\href`、`\url`、`\path`、`\verb`、`\lstinline` 都能显示且命令不丢失。
- 行内数学、行间数学、`equation`、`align`、`tabular`、`lstlisting`、`verbatim`、`thebibliography` 能显示，但不应被错误改写。
- 注释行和视觉空白命令不会把同一段正文切得过碎。

### 6.3 DOCX

打开 `testdoc/chunk-test.docx`。

重点检查：

- 可正常导入并切块。
- 简单 docx 可以进入编辑模式。
- `chunk-test.docx` 中的标题、列表、超链接、粗斜体下划线、公式、分页符、目录占位、图片占位、文本框占位、表格占位、分节符都应有稳定展示。
- 带常见行内样式、超链接、公式锁定、分页符占位、目录/图片/文本框/表格占位的 docx 应允许正常导入、审阅并 finalize 写回。
- 带常见行内样式、超链接、公式锁定、分页符占位、目录/图片/文本框/表格占位的 docx 应允许进入“编辑终稿”。
- 若纯文本编辑修改跨越原有行内样式边界，或改坏公式/分页符/占位符等锁定内容，保存时必须直接报错并拒绝写回。
- 编辑保存时必须保持段落数量不变；标题、列表、常见制表和单个段内换行应可正常写回。
- 应用修改后可执行 finalize；若原文件被外部改动，写回应直接失败，不得误覆盖。
- 遇到当前仍不支持安全写回的 docx 结构时，应在进入编辑或执行 finalize 前直接提示具体原因，不要等到用户改完再报。

模板专项回归再补一轮 `testdoc/04-3 作品报告（大数据应用赛，2025版）模板.docx`：

- 能正常导入，不因 `drawing`、`AlternateContent`、`bookmarkStart`、`proofErr`、`sdt` 直接报错。
- 正文、标题、列表正文、超链接显示文本仍按正常 chunk 展示。
- 图片显示为 `[图片]` 锁定占位。
- 图表显示为 `[图表]` 锁定占位。
- 普通图形或 SmartArt 类对象显示为 `[图形]` 锁定占位。
- 组合图形显示为 `[组合图形]` 锁定占位。
- 文本框显示为 `[文本框]` 锁定占位，文本框里的说明文字不能混入正文可编辑区。
- 目录显示为 `[目录]` 锁定占位。
- 表格显示为 `[表格]` 锁定占位。
- 这些占位符在工作区可见，但不会进入 AI 改写，也不能在编辑器里被改后安全写回。
- 无法归类且无法安全抓取的未知图形对象，应直接导入失败，不得伪装成正文或错误占位符。
- 对模板不做任何修改直接执行 finalize，应能原样写回成功。

以下 docx 边界建议通过 `cargo test docx -- --nocapture` 验证：

- 列表正文可见。
- 表格显示为占位符。
- 公式可见但锁定。
- 分页符显示为占位符。
- 超链接保留显示文本和目标 URL 元数据。
- 粗体/斜体/下划线等常见行内格式有 presentation。
- 嵌入 Office 对象显式报错。

### 6.4 PDF

打开 `testdoc/chunk-test.pdf`。

重点检查：

- 可导入、可改写、可导出。
- 不允许进入编辑模式。
- 不允许 finalize 写回覆盖原文件。

## 7. 失败信号

出现以下情况应直接判为缺陷：

- “处理所选”实际处理了未选中的块。
- 选择状态条跑回顶部工具栏，导致按钮或其他组件位移。
- PDF 进入编辑或允许 finalize。
- docx 在不满足写回条件时静默改坏原文件。
- 编辑页存在未保存修改时，仍允许直接切文件或开始优化。
- 切段策略在已有修改对/进度时仍可直接变更。
