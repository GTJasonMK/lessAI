# 审阅右栏列表化重构设计

## 背景

当前工作台右侧审阅区在布局上是一个窄栏，而不是一个可自由展开的主内容区。实际宽度约束已经写在布局样式中：

- 右栏宽度范围为 `300px - 420px`

在这个宽度下，现有审阅区同时承载了以下职责：

- 顶部统计信息
- `diff / 原文 / 候选稿` 切换
- 当前修改详情大卡片
- 修改记录列表
- 应用、忽略、删除等动作

这会导致两个问题：

1. 右栏在窄宽度下承担了过多层级，视觉噪声过高。
2. 左侧文档区本来已经具备“定位到对应 rewrite unit 并展示真实改动”的能力，右栏再重复展示大块 diff，职责发生重叠。

因此本次调整不再把右栏当成“修改详情面板”，而是收敛为“修改记录导航器 + 行内操作入口”。

## 目标

- 将右侧审阅区重构为单列紧凑列表。
- 移除右栏中的大块 diff / 原文 / 候选稿详情展示。
- 保留点击某条建议后，左侧文档区自动定位到对应 rewrite unit 的现有链路。
- 将右栏动作压缩为 hover 才出现的轻量操作，默认保持安静。
- 只复用当前项目已有的动作接口，不新增第二套审阅行为模型。

## 范围

本次仅调整右侧审阅栏在工作台模式下的展示和交互组织，包括：

- 审阅列表的行布局
- 行 hover 动作展示方式
- 行二级入口的动作归类
- 顶部固定区保留哪些摘要信息
- 右栏与左侧文档区的职责分工

本次不调整：

- 左侧文档区的 rewrite unit 渲染规则
- 后端 suggestion / rewrite unit / writeback 数据结构
- AI 改写流程
- 写回逻辑
- 编辑模式界面

## 当前链路结论

当前项目已经具备“右栏选中一条，左侧定位到正文对应位置”的闭环，不需要为这次重构新增定位机制：

- 右栏点击建议时，同时触发 `onSelectRewriteUnit(...)` 和 `onSelectSuggestion(...)`
- 文档流在 `activeRewriteUnitId` 变化后，会将对应 rewrite unit 节点 `scrollIntoView`

因此，右栏去掉 diff 详情后，用户仍然可以通过“点一条 -> 左侧定位 -> 直接看正文中的真实改动”完成审阅，不会丢失核心能力。

## 设计原则

### 1. 右栏只做索引和决策

右栏只负责回答两个问题：

- 当前我在处理哪一条修改对
- 我要对这条修改对执行什么动作

右栏不再负责完整承载正文内容。

### 2. 正文中的真实改动只看左侧

所有具体改动、上下文、保护区标记、rewrite unit 高亮，都交给左侧文档区原位展示。右栏不再提供第二份冗余正文。

### 3. 默认安静，悬浮再操作

在窄栏里，所有常驻按钮都会显著放大噪声。因此动作改为：

- 默认隐藏
- hover 到当前行时再显示

### 4. 主动作和次动作分层

高频操作直接显示，低频或语义更“撤销/例外”的动作收进二级入口。

## 最终方案

### 总体结构

右栏重构为两层：

1. 顶部轻摘要区
2. 下方单列记录列表

不再保留：

- 右栏大 diff 卡片
- `diff / 原文 / 候选稿` 切换
- 右栏底部大操作条

### 顶部轻摘要区

顶部只保留最低限度的全局信息，例如：

- 修改对总数
- 待审阅数量
- 当前条目序号

不在顶部常驻放置应用、删除、忽略、取消应用等按钮。

### 列表项结构

每条记录压缩为单行主信息 + 一行元信息，整体高度保持紧凑。

推荐字段：

- `#序号`
- 单行摘要标题
- 时间
- 当前状态

其中摘要标题只保留单行，超出部分尾部省略。

示例：

- `#12 自动汇总问卷结果，压缩后半句长度`
- `18:42 · 已应用`

### 行 hover 主动作

当鼠标悬浮在某条记录上时，直接露出两个主动作：

- `应用`
- `删除`

这两个动作是当前审阅里最直接、最明确、最常用的操作，因此作为一级操作直接显示。

### 二级入口

hover 时同时显示一个 `···` 二级入口，用于承接非主路径动作：

- `忽略`
- `取消应用`

其中显示文案随当前 suggestion 状态变化：

- 未应用时显示 `忽略`
- 已应用时显示 `取消应用`

## 接口映射

本次设计不引入新接口，只重排现有前端入口。

### 行点击

用途：

- 激活当前 suggestion
- 激活其对应 rewrite unit
- 触发左侧正文自动定位

映射：

- `onSelectRewriteUnit(rewriteUnitId)`
- `onSelectSuggestion(suggestionId)`

### 行 hover 主动作：应用

映射：

- `onApplySuggestion(suggestionId)`

### 行 hover 主动作：删除

映射：

- `onDeleteSuggestion(suggestionId)`

### 二级入口：忽略 / 取消应用

映射：

- `onDismissSuggestion(suggestionId)`

说明：

- 该接口在当前实现里既承担“忽略”也承担“取消应用”
- 具体文案由 suggestion 当前状态决定

### 失败重试

失败重试不再放在全局固定栏。

若某条 suggestion 对应的 rewrite unit 处于失败状态，则在该行的 `···` 二级入口中补一个：

- `重试`

映射：

- `onRetry()`

本次只调整入口位置，不改变失败重试逻辑本身。

## 与当前实现相比的删减项

重构后需要从右栏移除以下内容：

- 当前 suggestion 的 `diff-view`
- `reviewView` 对应的切换按钮
- `source` / `candidate` / `diff` 三态正文展示
- 顶部固定动作条中的 suggestion 操作按钮

保留：

- suggestion 排序与激活逻辑
- 当前 suggestion 高亮态
- 与左侧文档区的联动定位

## 视觉约束

- 列表项默认保持低噪声，避免像按钮墙。
- hover 动作只在当前行出现，不要让整列持续闪烁或出现多余动画。
- 单行摘要必须稳定省略，不允许挤压行内按钮导致布局抖动。
- `应用` 使用主动作视觉强调。
- `删除` 允许弱危险色，但不能做成压倒性的破坏性主按钮。
- `···` 应保持低存在感，只在 hover 时出现。

## 非目标

- 不支持在右栏直接阅读完整 candidate 文本。
- 不在本次中增加批量应用、批量删除、批量忽略。
- 不增加上一条 / 下一条独立动作接口。
- 不增加新的审阅模式切换。

## 风险与控制

### 风险 1：去掉右栏 diff 后，用户可能一时不适应

控制：

- 依赖现有“点击右栏条目 -> 左侧自动定位”的闭环
- 保证左侧当前 rewrite unit 的高亮足够明显

### 风险 2：行内 hover 动作过多，导致窄栏拥挤

控制：

- 一级动作严格限定为两个：`应用`、`删除`
- `忽略 / 取消应用` 进入二级入口

### 风险 3：删除作为主动作过于激进

控制：

- 删除按钮使用弱危险色
- 保持与应用同级，但不做最强视觉强调

## 测试

- 前端类型检查：`pnpm run typecheck`
- 前端样式回归：`node scripts/ui-regression.test.mjs`
- 手动验证以下交互：
  - 点击某条 suggestion 后，左侧文档区定位到对应 rewrite unit
  - hover 某条 suggestion 时，只出现该行的 `应用 / 删除 / ···`
  - 二级入口能根据状态显示 `忽略` 或 `取消应用`
  - 删除、应用、忽略/取消应用后的激活态和列表状态能正确刷新
